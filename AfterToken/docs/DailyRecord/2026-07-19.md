## 工作内容

1. **修复 PlayerDeathUI 按钮无响应（根因：UIModule userDatas 被污染）**
   - 现象：玩家死亡弹窗的 Restart / Back to Lobby 点击无反应。MCP 现场取证：两个按钮 `runtimeListeners=0`，即从未绑定。
   - 根因：`UIModule.ShowUIAsync<T>(userDatas)` 与 `ShowUIAsync(Type, userDatas)` 内部调用写成了 `ShowUIImp<T>(true, userDatas)` / `ShowUIImp(type, true, userDatas)`，但 `ShowUIImp` 只有 `(params object[])` 签名，多余的 `true` 被 params 打包进 `_userDatas[0]`。`PlayerDeathUI.BindEvents` 校验 `_userDatas[0] is PlayerDeathHandler` 永远为 false，跳过绑定。
   - 修复：`UIModule.cs` 两处调用移除残留的 `true` 实参（`UIModule.cs:265`、`UIModule.cs:276`）。
   - 影响面：所有通过 `ShowUIAsync` 传 userDatas 的窗口此前都拿不到正确数据（当前仅 PlayerDeathUI 使用）；`LogUI` 读取 `UserData` 此前同样只能拿到 `true`，一并恢复正常。
   - 排查手段备忘：MCP `execute_code` 反射读取 `_uiStack`/按钮 `m_RuntimeCalls`，证实绑定数为 0。

2. **修复 SettingsUI 全部控件无响应（根因：节点绑定路径缺前缀）**
   - 现象：设置面板 Close 按钮（及灵敏度滑条）无反应。
   - 根因：`UIBase.FindChildComponent` 基于 `transform.Find`（**不递归**，支持 `a/b` 路径），而 SettingsUI 的控件都在 `m_rect_ContentRoot` 子节点下，`ScriptGenerator` 里写的是裸节点名（`m_btn_Close` 等），查找不到返回 null，绑定被 `if (x != null)` 静默跳过。
   - 修复：`SettingsUI.cs` 的 `ScriptGenerator` 三个路径补全为 `m_rect_ContentRoot/m_slider_Sensitivity`、`m_rect_ContentRoot/m_text_SensitivityValue`、`m_rect_ContentRoot/m_btn_Close`。
   - 教训：UI 脚本内节点路径必须与 prefab 实际层级一致（与 UI_STANDARDS 一致）；新增 UI 时注意生成工具若调整过层级需同步更新路径。

5. **修复关闭设置 UI 后光标不隐藏（系统光标与游戏准星并存）**
   - 现象：战斗中关闭 SettingsUI 后，屏幕上同时存在系统鼠标与游戏准星；预期只保留准星（OS 光标应隐藏并锁定）。
   - 根因：Windows 下 `Cursor.visible = false` / `lockState = Locked` 在窗口未捕获鼠标时会被忽略（`docs/modules/ui/cursor-system/README.md` 已记录的已知问题），而 `CursorManager.ApplyCursorState` 隐藏分支只有「延迟一帧单次重试」，不足以覆盖；`Cursor.visible`/`lockState` 读回的是 Unity 侧标志而非 OS 真实状态，无法据此判断生效。
   - 修复：`CursorManager.cs` 隐藏分支改为约 0.5 秒逐帧无条件重设隐藏/锁定（30 帧），期间任何新的显示请求取消重试；与既有 focusChanged 焦点补偿共同构成保障。
   - 同步更新 `docs/modules/ui/cursor-system/README.md` 的解决方案与代码要点。
   - 附带发现（未处理）：Console 有 `DontDestroyOnLoad only works for root GameObjects` 警告（某非根对象被 DDOL）；传送门 DestinationLabel 出现中文字符缺字（LiberationSans 静态字库 Latin-only，应按英文约束改配置）。

7. **修复死亡时按传送门导致新场景冻结（死亡/传送判定）**
   - 现象：玩家死亡瞬间在传送门触发区内按 E，会被传送到目的地，但新场景完全无法操作。
   - 根因：死亡时 `PlayerDeathHandler` 执行 `GamePauseManager.PushTimeScale(0)` 暂停游戏，而全项目只有 `ConfirmRestart` / `ConfirmReturnToLobby` 两个出口会 `PopTimeScale`；传送门交互（`PortalSystem.OnInteractPressed`）无死亡检查，绕过死亡确认直接切流程，暂停请求泄漏到新场景（`timeScale` 恒为 0）。
   - 修复（双层）：
     1. `PortalSystem.OnInteractPressed` 增加死亡判定：玩家已死亡（`PlayerEntity.IsDead`）时忽略交互，必须走死亡确认。
     2. `GamePauseManager` 新增 `Reset()`（清空全部请求并恢复 `timeScale=1`），`GameApp.ChangeProcedure<T>()` 每次流程切换时调用，兜底防止任何暂停状态跨流程泄漏（含武器轮盘慢动作等其他路径）。
   - 同步更新 `docs/modules/scene/portal-system/README.md`（新增「死亡与传送的判定」一节）。
   - 验证：编译 0 error / 0 warning；复现场景（传送门旁死亡按 E → 应无反应；Restart 后新战斗时间流动正常）交由人工 Play Mode 确认。
