# 俯视角 2D 射击游戏 + 模拟经营 — 项目架构方案

> 基于 TEngine（HybridCLR + YooAsset + UniTask + Luban）框架，采用 **纯 OOP + 对象池 + 事件解耦** 的架构设计。包含射击战斗和模拟经营两大玩法模块，通过共享玩家数据层连接。

---

## 一、架构设计目标

1. **低耦合**：系统间通过事件通信，避免直接引用
2. **高内聚**：每个系统职责单一，内部可直接调用
3. **可热更**：所有 gameplay 代码放在 `GameLogic` 热更程序集
4. **可扩展**：新增武器、敌人、关卡、建筑、生产配方只需增加配置和少量代码
5. **性能可控**：对象池 + 统一更新 + 分帧 + Atlas 合批
6. **玩法隔离**：射击和经营模块独立，通过共享层通信

---

## 二、程序集划分

| 程序集 | 路径 | 热更 | 职责 |
|--------|------|------|------|
| `Assembly-CSharp` | `Assets/GameScripts/` | ❌ | `GameEntry`、主包流程 `Procedure/` |
| `Launcher` | `Assets/Launcher/` | ❌ | 启动器 UI、资源更新流程 |
| `TEngine.Runtime` | `Assets/TEngine/Runtime/` | ❌ | 框架核心模块 |
| `GameProto` | `Assets/GameScripts/HotFix/GameProto/` | ✅ | Luban 配置表生成代码 |
| `GameLogic` | `Assets/GameScripts/HotFix/GameLogic/` | ✅ | 所有游戏业务逻辑 |

**依赖规则**：`GameLogic → GameProto → TEngine.Runtime`，禁止反向依赖。

## 六、系统间通信方式

### 6.1 跨系统通信：GameEvent 接口事件

所有系统之间通过 `GameEvent.Get<T>()` 发送事件，接收方通过 `GameEventMgr` 监听。

```csharp
// 发送方
GameEvent.Get<IBattleEvent>().OnEntityDamaged(damageInfo);

// 接收方
private readonly GameEventMgr _eventMgr = new();

private void Awake()
{
    _eventMgr.AddEvent<DamageInfo>(IBattleEvent_Event.OnEntityDamaged, OnEntityDamaged);
}

private void OnDestroy()
{
    _eventMgr.Clear();
}
```

### 6.2 UI 监听业务事件：AddUIEvent

```csharp
[Window(UILayer.UI, "BattleMainUI")]
public class BattleMainUI : UIWindow
{
    protected override void RegisterEvent()
    {
        AddUIEvent<int, int>(IPlayerEvent_Event.OnHpChanged, OnHpChanged);
        AddUIEvent<int, int>(IPlayerEvent_Event.OnAmmoChanged, OnAmmoChanged);
    }
}
```

### 6.3 系统内部通信：直接方法调用

同一系统内部允许强引用，保持高内聚。

```csharp
public class PlayerSystem
{
    private PlayerEntity _playerEntity;
    
    public void TakeDamage(int damage)
    {
        _playerEntity.Health -= damage;
        UpdateUI();
    }
    
    private void UpdateUI() { /* 内部调用 */ }
}
```

### 6.4 系统与 Entity：Entity 通过事件回调或 System 引用

Entity 不直接持有其他 Entity 的引用，而是通过事件通知 System。

```csharp
public class ProjectileEntity : MonoBehaviour
{
    private void OnTriggerEnter2D(Collider2D other)
    {
        // 只检测，具体逻辑交给 ProjectileSystem / BattleSystem
        GameEvent.Get<IProjectileEvent>().OnProjectileHit(Id, other.gameObject);
    }
}
```

```
    ┌────────────────────────────────────────────┐
    │              共享层（Shared）                │
    │  PlayerProfileSystem / CurrencySystem      │
    │  InventorySystem / RewardSystem            │
    └────────────────────────────────────────────┘
                 │

        [SimulationMainUI] / [OrderSystem]
                 ▲
                 │
        [SimulationSystem] ──► [BuildingSystem]
                 │                  │
                 ▼                  ▼
        [SimTimeSystem]      [ProductionSystem]
                 │                  │
                 ▼                  ▼
        [WorkerSystem]      [FarmSystem]

**跨玩法通信示例**：
- `BattleSystem` → `IBattleResultEvent.OnBattleWin` → `RewardSystem` → `CurrencySystem` / `InventorySystem`
- `OrderSystem` → 消耗物品 → `InventorySystem` → 发放奖励 → `CurrencySystem`
- `ProductionSystem` → 产出物品 → `InventorySystem` → `IInventoryEvent.OnItemChanged`
```

**关键依赖规则**：
- 箭头方向表示“监听事件”或“调用”
- 不允许循环依赖：A 系统不直接监听 B 系统，同时 B 系统又直接监听 A 系统
- 所有跨系统依赖通过事件接口抽象

## 九、对象池架构

### 9.1 统一池接口

```csharp
public interface IGameObjectPool
{
    GameObject Get(string key);
    void Recycle(string key, GameObject go);
    void Preload(string key, int count);
    void Clear(string key);
}
```

### 9.2 对象池管理器

```csharp
public class PoolSystem : MonoBehaviour
{
    private readonly Dictionary<string, GameObjectPool> _pools = new();
    
    public GameObject Get(string key, GameObject prefab, Transform parent)
    {
        if (!_pools.TryGetValue(key, out var pool))
        {
            pool = new GameObjectPool(prefab, parent);
            _pools[key] = pool;
        }
        return pool.Get();
    }
    
    public void Recycle(string key, GameObject go)
    {
        if (_pools.TryGetValue(key, out var pool))
        {
            pool.Recycle(go);
        }
        else
        {
            Destroy(go);
        }
    }
}
```

### 9.3 需要池化的对象

| 对象类型 | 池 Key 示例 | 说明 |
|---------|------------|------|
| 普通子弹 | `Projectile_Normal` | 数量最多，必须池化 |
| 敌人 | `Enemy_Slime` / `Enemy_Zombie` | 按类型分池 |
| 特效 | `Effect_MuzzleFlash` | 短生命周期 |
| 掉落物 | `Pickup_Gold` / `Pickup_Health` | 数量中等 |
| 伤害数字 | `DamageText` | UI 对象池 |

## 十三、解耦策略总结

| 策略 | 实现方式 | 解决的问题 |
|------|---------|-----------|
| **事件总线** | `GameEvent` 接口事件 | 系统间解耦 |
| **配置驱动** | Luban 配置表 | 数据与逻辑分离 |
| **对象池** | `PoolSystem` | 避免 Instantiate/Destroy 耦合 |
| **服务定位器** | `GameModule.xxx` | 统一访问框架能力 |
| **依赖注入** | `GameApp` 初始化系统 | 避免单例滥用 |
| **接口隔离** | 每个系统有明确接口 | 降低变更影响范围 |
| **数据与表现分离** | `Entity` 只负责表现，`System` 负责逻辑 | 便于单元测试和热更 |

## 十五、禁忌清单

| ❌ 禁止 | ✅ 正确做法 |
|--------|-----------|
| 系统间直接 `FindObjectOfType` | 通过 `GameEvent` 通信 |

| 每个子弹一个 `Update()` | `ProjectileSystem` 统一 `Update` |
| 系统间循环引用 | 通过事件接口解耦 |
| UI 直接修改玩家数据 | UI 发送事件，`PlayerSystem` 处理 |
| 硬编码数值 | 使用 Luban 配置表 |
| 热更代码引用主包 internal 类型 | 遵守程序集依赖规则 |
